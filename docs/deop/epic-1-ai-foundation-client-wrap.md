## Task 1.2.1 — IAiClient Interface & Implementation

Defined `IAiClient.GenerateAsync(request) → result` as the single entry point every backend AI call must flow through, together with the supporting record types (`AiGenerateRequest`, `AiGenerateResult`, `AiUsage`, `AiGenerateMetadata`, `AiMessage`, `ModelClass`) and a typed `AiClientException` so callers can decide between graceful fallback and surfacing the error. The first concrete implementation, `AnthropicAiClient`, talks to the Anthropic Messages API through a thin `IAnthropicMessagesTransport` (typed `HttpClient`, configured from `Ai:BaseUrl` / `Ai:ApiKey` / `Ai:AnthropicVersion` / `Ai:RequestTimeoutSeconds`).

The cross-cutting concerns called out by the scope — cost translation, content filtering, retry, output validation — live in a provider-agnostic pipeline (`AiClientBase`) with seam interfaces (`IAiCostCalculator`, `IAiContentFilter`, `IAiRetryPolicy`, `IAiOutputValidator`, `IAiModelResolver`). `AiCostCalculator` and `AiModelResolver` ship as working config-driven implementations; filter/retry/validator ship as pass-through defaults that Tasks 1.2.2, 1.2.3, and 1.2.4 will replace by swapping their DI registration — no pipeline code changes. This is the same seam the Anthropic-direct → Azure AI Foundry provider migration will use post-launch.

Verified end-to-end with `dotnet build SSP.Core.sln` clean and 33/33 unit tests green, covering cost calculation, model resolution, request mapping, content-filter reject/redact paths, validator rejection, transport-exception wrapping as `TransientExhausted`, retry-policy invocation, cancellation propagation, and DI registration. Known gaps deferred to follow-up tasks: real retry (1.2.2), schema-based validation (1.2.3), PII / banned-content filter (1.2.4), `ai_usage` persistence (Story 1.3).

## Task 1.2.2 — Retry & Error Handling

Replaced the pass-through retry default with `PollyAiRetryPolicy`, a Polly v8 (`Polly.Core` 8.5.0) `ResiliencePipeline` that applies exponential backoff with jitter to every provider call. A new internal `AiRetryClassifier` decides what gets retried: `HttpRequestException` with status 408/429/5xx or no status, `TimeoutException`, and `TaskCanceledException` are transient; everything else (4xx other than 408/429, generic exceptions) is permanent and short-circuits without burning the budget. The retry budget (`MaxAttempts`, `InitialDelayMilliseconds`, `MaxDelayMilliseconds`) is sourced from a new `Ai:Retry` section in `appsettings.schema.json` / `appsettings.common.json`, rebuilt when `IOptionsMonitor` fires, and falls back to repo defaults (3 / 200ms / 5000ms) when the section is omitted.

Permanent failures and exhausted-transient failures flow into a new `IAiFallbackResponder` seam. The default `NoFallbackAiResponder` returns null and the pipeline rethrows as `AiClientException` — features that have not opted in still see real errors, satisfying the "errors are never swallowed" rule. The shipped `TemplateAiFallbackResponder` reads per-feature templates from a new `Ai:Fallback` array; when a template is present, `AiClientBase` returns a degraded `AiGenerateResult` with `Metadata.IsFallback = true`, a `FallbackReason` of `transient-exhausted` or `provider-permanent`, zero token usage, and `StopReason = "fallback"`. `AiGenerateMetadata` gained two optional fields (`IsFallback`, `FallbackReason`) so Story 1.3's `ai_usage` writer can attribute these calls correctly without a separate code path.

Verified with `dotnet build SSP.Core.sln` clean and 62/62 AI unit tests green — covering the transient-vs-permanent classification matrix, retry-then-recover, retry-budget exhaustion, permanent-failure short-circuit, cancellation propagation, template lookup (hit, miss, empty table, case-insensitive), and the new pipeline branches for both permanent-with-fallback (returns template) and permanent-without-fallback (still throws). Structured logging is wired at every boundary: `Warning` on each retry attempt with the next delay, `Error` on final failure with the exception type and classified `Kind`, and `Information` on fallback application. The retry policy is still a single seam, so the future Anthropic-direct → Azure AI Foundry migration only needs to swap the provider call — the budget, classifier, and fallback table stay in place.

## Task 1.2.3 — Output Validation

Stood up a schema-driven output-validation seam built around FluentValidation. The new `AiOutputSchema<TSchema>` deserialises the model's JSON response into a strongly-typed feature record and runs an `AbstractValidator<TSchema>`; parse failures, null payloads, and rule failures all surface as `AiOutputValidationResult.Errors` so the pipeline can react instead of crashing. Features opt in by calling `services.AddAiOutputSchema<TSchema, TValidator>(featureId)`, which feeds an `AiOutputSchemaRegistration` into the default `AiOutputSchemaRegistry`. The previous pass-through `IAiOutputValidator` has been replaced by `SchemaAiOutputValidator`, which routes by feature: registered features run the schema, unregistered features short-circuit to `Valid` so call sites that have not yet declared a contract keep working.

`AiClientBase.GenerateAsync` now wraps the provider call, output filter, and validator in a corrective-retry loop bounded by a new `Ai:OutputValidation:MaxAttempts` config knob (default 2). On a validation failure, the pipeline replays the rejected output as an assistant turn and appends a corrective user message — built by `AiCorrectivePrompt.Build` from the validator's error list — then re-invokes the provider with the augmented conversation. Input filtering still runs only once; the loop touches only the steps that depend on the latest provider output. When the attempt budget is exhausted the pipeline emits a structured `Error` log and throws `AiClientException` with `Kind = ValidationFailed` and a message that includes the attempt count and the last set of errors, matching the scope's "After 2 failures → typed error return" contract.

Verified with `dotnet build SSP.Core.sln` clean and 78/78 AI unit tests green — adding coverage for JSON parse failures, null/empty payloads, FluentValidation property-scoped errors, registry lookup (hit, miss, case-insensitive), DI registration of `AddAiOutputSchema<,>`, corrective-retry success on the second attempt with assertions on the replayed messages, validation-loop exhaustion across both default and explicitly-configured `MaxAttempts`, and the invariant that the input filter runs once even when the loop iterates. The provider-failure fallback path from Task 1.2.2 is untouched: provider exceptions still short-circuit the loop and route through `HandleProviderFailureAsync`, so retry-vs-validation concerns stay layered rather than entangled.

## Task 1.2.4 — Content Filtering

Replaced the pass-through content filter with `ConfigurableAiContentFilter`, a real `IAiContentFilter` driven by a new `Ai:ContentFiltering` section in `appsettings.schema.json` / `appsettings.common.json`. The implementation composes two collaborators: a static `PiiDetector` that scans for emails, phone numbers, US SSNs, credit-card-shaped sequences, and IPv4 addresses with stable `[PII:<kind>]` redaction placeholders; and a config-driven `BannedTermDetector` that reads operator-supplied terms, escapes regex metacharacters, compiles a case-insensitive word-bounded match per term, classifies hits by `BannedSeverity` (low/medium/high), and refreshes its cache when `IOptionsMonitor<AiConfig>` fires.

Behaviour is governed by per-feature thresholds — the customer-configurable knob the scope asked for. Each feature can independently override `PiiInputAction`, `PiiOutputAction`, `BannedInputAction`, `BannedOutputAction`, and `BannedMinSeverity`; features without an entry inherit the repo defaults (input PII off, output PII redact, input banned reject, output banned redact, minimum severity medium). `PiiEnabled` and `BannedEnabled` act as global kill-switches. Input redaction is intentionally treated as "off" — mutating the caller's messages is outside the current seam contract — while the output side honours redact and reject equally. Detection events surface as structured `Information` logs with feature, direction, action, hit count, and a kind/category summary; the matched substrings themselves are never logged, so the redaction trail is auditable without leaking what was being protected.

Verified with `dotnet build SSP.Core.sln` clean and 104/104 AI unit tests green — adding coverage for each PII pattern (and the deliberately strict phone separator rule), banned-term word boundaries, severity threshold filtering, regex-metacharacter escaping of operator-supplied terms, feature-override paths in both directions (reject on input, reject vs. redact on output, full opt-out), and the two global kill-switches. Story 1.3's `ai_usage` writer will consume the same log events for `content_filter_outcome`; no other code paths are affected by this swap.

## Task 1.3.1 — ai_usage Schema Migration

Created the `ssp.ai_usage` table as the persistence target for Story 1.3's mandatory per-call logging. Scope §6.1 lists the table as an alter, but the Sweetspot tenant DB had no prior `ai_usage` — this is the table's first creation. The new `AiUsageLog` entity (`SSP.Database.Tenant/Entities/AiUsageLog.cs`) covers the §1.3 + §6.1 column set: `id`, `account_id` (mapped from the strategy's `org_id` per Sweetspot convention), `user_id`, `feature`, `model`, `input_tokens`, `output_tokens`, `cost_usd` (`numeric(18,6)`), `latency_ms`, `success`, `error_message`, `prompt_hash`, `output_hash`, `consent_status`, `data_residency_region`, `trace_id`, `content_filter_outcome` (`jsonb`), and `created_at`. The class is named `AiUsageLog` to avoid clashing with the existing in-memory `SSP.AI.Client.Models.AiUsage` record returned by `IAiClient`; the table name stays `ai_usage`.

FK delete behaviour and indexes match the tenant-DB conventions in `Core/Database/AGENTS.md`. `account_id` is NOT NULL with `ON DELETE CASCADE` so that GDPR right-to-erasure on an account wipes its AI history; `user_id` is nullable with `ON DELETE SET NULL` so system / background calls can still log and a deleted user does not orphan rows. Three indexes were added: `ix_ai_usage_account_id` (FK companion), `ix_ai_usage_account_id_created_at` (per-org per-month cost rollups for §12.1), and `ix_ai_usage_trace_id` (single-action audit linkage for §12.5). `content_filter_outcome` is `jsonb` so the structured outcome already produced by `ConfigurableAiContentFilter` (Task 1.2.4) — PII categories, banned-term hits with severity, action taken — can be persisted verbatim for AI Act explainability per §13.7.

Generated via `..\Tools\CreateMigration.ps1 AddAiUsageTable` from `Core/Database/SSP.Database.Tenant/`, producing `20260608155642_AddAiUsageTable.cs`, its designer file, the updated `TenantDbContextModelSnapshot.cs`, and the idempotent SQL delta at `Migrations/Sql/AddAiUsageTable.sql`. The SQL is vendor-neutral Postgres (no Supabase-specific syntax), guarded by `IF NOT EXISTS(SELECT 1 FROM "__EFMigrationsHistory" …)` per the helper's idempotent template. Verified with `dotnet build src/backend/Core/SSP.Core.sln` clean (0 errors; only pre-existing CS1591 / CS8625 / CS0618 warnings) and 104/104 AI unit tests still green — schema-only change, so no new unit tests added. ## Task 1.3.2 — Usage Log Service

Introduced `IAiUsageLogger` as the single persistence seam for `ssp.ai_usage` and wired it into the orchestration pipeline so every `IAiClient.GenerateAsync` call — success, fallback, or thrown failure — produces exactly one audit row. The new `SSP.AI/Usage/` folder ships four files: `IAiUsageLogger` (single-method seam), `AiUsageLogEntry` (provider-agnostic DTO carrying every column value), `AiUsageLogger` (default implementation mirroring `AuditService.CreateAuditEventInDbSafelyAsync`'s synchronous best-effort pattern via `ITenantDbContextFactory.CreateForAccount`), and `AiUsageHasher` (SHA-256 hex utility producing `prompt_hash` from `system + role:content` lines and `output_hash` from the filtered output). `AiGenerateRequest` grew four fields — `AccountId` (required), `UserId`, `TraceId`, `ConsentStatus` — so BFF callers extract from `HttpContext.User` claims via the existing `ClaimsPrincipalExtensions` and pass them explicitly, matching the convention used by `AuditService`. No new middleware, no implicit DI context.

`AiClientBase.GenerateAsync` was refactored around an outer `try/catch/finally`: the success and fallback returns capture a local `AiGenerateResult? result`, the catch records the failing exception, and the finally builds an `AiUsageLogEntry` (model resolved from result metadata or the pre-resolution model id, region from `AiGenerateMetadata.Region` then `AiConfig.Region`, fallback markers surfaced via `BuildErrorMessage`) and awaits `_usageLogger.LogAsync(entry, CancellationToken.None)` so a cancelled upstream still produces the row. The logger call is itself wrapped in try/catch — a DB hiccup is logged at `Error` via `ILogger` but never propagates. `ValidateRequest` gained an `AccountId == Guid.Empty` check (scope §7.2 — no AI call without an org). The strategy's "cost calculation service (per-provider pricing tables)" subtask is already satisfied by Task 1.2.1's `IAiCostCalculator` + `Ai:CostTable`; this task consumes the value via `result.Usage.CostUsd` and writes nothing new. `content_filter_outcome` stays null in 1.3.2: surfacing the structured outcome from `ConfigurableAiContentFilter` (Task 1.2.4) requires changing the filter contract and is deferred to a follow-up.

Verified with `dotnet build src/backend/Core/SSP.Core.sln` clean (0 errors; warnings unchanged from the pre-existing CS1591 / CS8625 / CS0618 baseline) and **120/120** AI unit tests green — up from 104 with new coverage in `AiUsageHasherTests` (hex format, determinism, ordering and system-prompt sensitivity), `AiUsageLoggerTests` (factory routed by AccountId, row mapped through, factory-throw swallowed; uses `Microsoft.EntityFrameworkCore.InMemory` and a stub `ITenantDbContextFactory` with deterministic database names so the test can read back via a fresh context after the logger disposes its write context), and `AnthropicAiClientTests` (success row with mandatory fields, minted-trace-id default, "unknown" consent default, validation-exhausted writes `success=false` with `ValidationFailed`, fallback row records zero tokens + `fallback:transient-exhausted`, input-filter reject still writes a row with no output hash, and a `ThrowingUsageLogger` confirms a logger failure does not poison the AI result). The `Usage/` folder is now wired end-to-end; the BFF callers that will adopt the new request fields land in Task 1.5.1 (email-generate migration).

## Task 1.2.5 — Azure AI Foundry Provider Adapter

Added Azure AI Foundry as a second provider adapter behind the existing `IAiClient` contract without changing feature callers. The new `SSP.AI/Client/AzureFoundry/` implementation translates the provider-neutral request into a chat-completions request, treats the resolved model id as the Foundry deployment name, maps system instructions into a `system` message, and maps Foundry response content, stop reason, trace id, and token usage back into `AiProviderResponse`. All retry, validation, content filtering, usage logging, cost calculation, spend-cap hooks, and graceful fallback behavior remains in `AiClientBase`, so Foundry inherits the same controls Anthropic-direct already uses.

Provider selection now happens in DI from `Ai:Provider` / `AI_PROVIDER`: `anthropic` remains the default and existing launch path, while `azure-foundry` (or `foundry`) resolves `AzureFoundryAiClient`. This keeps the task in provider-readiness / post-launch status unless Foundry is explicitly promoted to a launch dependency. `ModelClass` now includes `Opus` for reasoning-heavy workflows, `AiModelResolver` maps Sonnet / Haiku / Opus to `Ai:Models`, and `appsettings.common.json` / `appsettings.schema.json` now include Opus plus provider-specific cost-table rows for the placeholder Foundry deployment ids (`foundry-sonnet-deployment`, `foundry-haiku-deployment`, `foundry-opus-deployment`). Production environments must replace those placeholders with the actual Foundry deployment names and confirmed pricing before enabling `AI_PROVIDER=azure-foundry`.

## Task 1.4.1 — Context Block Base

Established the shared base abstractions every AI context block (brand, org, industry, methodology, audience, plan, insights, knowledge, assets, field-semantics) will implement. No concrete loader ships in this task — `BrandContext`, `OrgContext`, and `IndustryContext` land in Tasks 1.4.2, 1.4.3, and 1.4.4 on top of this base. The new `SSP.AI/Context/` folder contains three files. `IAiContext` declares the two members every context return type carries: an `Exists` flag (the "empty stub" signal — loaders never throw on missing or sparse source data, they return an empty stub with `Exists = false`) and a `ToPromptString()` method that returns the prompt-facing segment for the block. `ToPromptString()` must include only fields the LLM should reason about; renderer-only fields (brand colors, fonts, logos, footer HTML) are exposed as typed properties on the context object and stay out of the returned string (scope §2.14). When `Exists = false`, `ToPromptString()` returns `string.Empty` so callers can unconditionally concatenate segments.

`IContextBlock<TContext>` is the loader interface: `LoadBlockContextAsync(accountId, options?, cancellationToken)` → `TContext`, with `TContext : IAiContext` so feature code can compose multiple blocks via the shared contract without per-block branching. The interface XML doc pins the three scope §2 block rules: graceful degradation (missing or sparse source data must return an empty stub, not throw), no side effects (reading and formatting only — the AI client wrapper is the only writer), and cross-block composition allowed (a block may depend on other blocks via constructor injection). `AiContextOptions` is the shared caller-hint record carrying the two fields genuinely common across blocks — optional `UserId` (per-user knowledge scopes, audit linkage) and optional `Feature` (per-feature payload variation). Blocks that need narrower hints (knowledge top-K, audience id, asset id list) derive their own record from this base rather than adding fields here. `AccountId` is required at the interface level — passing `Guid.Empty` is contractually invalid (scope §7.2 forbids any AI substrate read not scoped to an account); concrete loaders will enforce this in 1.4.2 – 1.4.4.

The prototype-loader-to-Sweetspot-context mapping from the `ai-arch-scope.md` handover alignment section was reproduced in `docs/deop/task-1.4.1-context-block-base.md` so the mapping lives next to the substrate that implements it: `loadBrandContext → BrandContext` (split prompt tonal fields from renderer visual fields), `loadOrgContext → OrgContext`, `loadIndustryContext → IndustryContext` (deterministic defaults when no industry config exists), `loadProfileContext → BrandContext + future KnowledgeContext`, `loadContactFieldsContext → future FieldSemanticsContext`, `loadKnowledgeContext → future KnowledgeContext`, asset library access → future `AssetsContext` (enriched metadata only, semantic search waits for embeddings), `industry-examples → frontend/helper data`. The doc also captures the Phase 1 Option A substrate ownership stance (substrate lives inside Sweetspot .NET, context loaders read tenant data through internal EF Core-backed services — no external `GET /brand/context` contract until Deop confirms it needs a separate Node consumer) and the scope §2.14 / §2.15 / §2.16 invariants (brand visual fields renderer-only, assets enriched metadata only, semantic asset search deferred).

Verified with `dotnet build src/backend/Core/SSP.Core.sln` clean (0 errors; warnings unchanged from the pre-existing baseline) and **154/154** AI unit tests green — up from 147 with seven new tests in `AiContextConventionTests` exercising the contract via an in-test fake (`FakeContextBlock` + `FakeContext`). Coverage: empty stub returns `Exists = false` and `ToPromptString() == ""`; a populated context renders prompt-facing fields and excludes the renderer-only field (pinning the "no visual fields in prompt" invariant); `LoadBlockContextAsync` returns an empty stub on a missing-data path without throwing; `LoadBlockContextAsync` returns a populated context on a hit path; cancellation is propagated through `LoadBlockContextAsync`; `options: null` is accepted; `AiContextOptions` is derivable so block-specific options (e.g. a `FakeKnowledgeOptions` with `TopK`) extend the base record. No other SSP.AI code is modified by this task — the base unblocks Tasks 1.4.2 (Brand), 1.4.3 (Org), and 1.4.4 (Industry) without touching the client wrapper, usage logger, or provider adapters.

## Task 1.4.2 — Brand Context Block

First concrete context block built on the Task 1.4.1 base. Ships the typed `BrandContext` shape required by the handover alignment, the Phase 1 EF Core-backed read path through an `IBrandKitReader` seam, and the loader that transforms source data into the public context record. The actual `brand_settings` schema is intentionally out of scope — Phase 1 ships the typed shape and the read seam so the eventual schema-arrival commit is a single-file change in `EfCoreBrandKitReader`.

`BrandContext` (public `record`, implements `IAiContext`) carries the two cleanly separated responsibilities the handover requires. The prompt-facing tonal block — `VoiceDescription`, `ToneTags`, `Values`, `OneLiner` — are the only fields `ToPromptString()` renders, under a `## Brand Voice` header with one line per present field. The renderer-facing visual block — `Colors`, `ResolvedColorRoles`, `Fonts`, `Logos`, `EmailFooterHtml`, `TemplateBanners` — are exposed as typed properties for downstream renderers (the email template engine that lands in PR 10) and are forbidden from `ToPromptString()` per scope §2.14. `BrandContext.Empty` is the shared empty stub (`Exists = false`, all renderer collections present-but-empty, all strings null) so renderer code never needs to null-check a visual field. `ToPromptString()` returns `string.Empty` whenever `Exists = false` or every tonal field is empty — including the legitimate state where a brand row exists but the operator has only filled colors so far, so callers can detect "nothing to inject" without inspecting individual fields.

`IBrandKitReader.ReadAsync(accountId, ct) → BrandKitData?` is the Phase 1 Brand Kit read contract. It is **public** so a future Option B path (Deop-owned Node consumer via HTTP) can replace the EF Core implementation with a single DI swap; the AI substrate's only stable contract above this seam stays `IContextBlock<BrandContext>` for consumers. `EfCoreBrandKitReader` is the Phase 1 implementation: depends on `ITenantDbContextFactory` (same pattern as `AiUsageLogger`), returns `null` for every account today because the `brand_settings` schema has not landed. The dependency on the tenant factory is declared already so DI verification tests see the production shape, and so the schema-arrival commit only needs to add the actual EF Core query inside `ReadAsync`. `BrandContextBlock : IContextBlock<BrandContext>` validates `accountId != Guid.Empty` (scope §7.2 — no AI substrate read without an account), calls the reader, and projects `BrandKitData? → BrandContext`. The block is best-effort by design: a reader exception is logged at `Warning` and degrades to `BrandContext.Empty` so a transient tenant-DB hiccup never crashes the AI call. `OperationCanceledException` is the one exception that always propagates so an upstream cancel terminates the load immediately rather than silently degrading. DI registrations were added to `ServiceCollectionExtensions.AddAiServices()` as singletons: `IBrandKitReader → EfCoreBrandKitReader` and `IContextBlock<BrandContext> → BrandContextBlock`.

Verified with `dotnet build src/backend/Core/SSP.Core.sln` clean (0 errors; warnings unchanged from the pre-existing baseline) and **170/170** AI unit tests green — up from 154 with sixteen new tests across three brand test classes. `BrandContextTests` (8 tests) exercises the typed shape: empty stub invariants, the non-null empty renderer collections, `IAiContext` implementation, full-render path, **the renderer-exclusion invariant** (a populated context with hex values, font families, logo URLs, banner URLs, and footer HTML asserted absent from the prompt), partial tonal rendering, the "exists but tonal empty" branch returning `""`, and skipping blank list entries. `BrandContextBlockTests` (6 tests) exercises the loader: `Guid.Empty` throws `ArgumentException` before the reader is touched, reader-null produces the empty stub, reader-data flows every field through (with the prompt-vs-renderer split asserted again on the output), reader-throws degrades to the empty stub via the graceful-degradation path, cancellation propagates as `OperationCanceledException`, and `options: null` is accepted. `EfCoreBrandKitReaderTests` (2 tests) pins the Phase 1 behavior: the reader returns `null` for any account **and does not open a tenant DbContext** (the test factory throws `InvalidOperationException` if it does — explicit guard against premature schema coupling), and cancellation propagates. ## Task 1.4.3 — Org Context Block

Second concrete context block built on the Task 1.4.1 base. Loads Sweetspot account / organisation identity (organisation name, industry / sub-industry keys, postal address, country) directly from the existing `Account` entity via `ITenantDbContextFactory.CreateForAccount(accountId)` (handover §"AI substrate ownership" Option A; scope §2.5). Unlike the brand block, no schema arrival is gated on this work — `Account` is already populated for every tenant. **No `IOrgKitReader` seam is introduced**: the read is a single EF Core statement with one navigation include, and adding an interface would be ceremony without buying any of the things a seam usually does (isolating schema churn, enabling alternative providers, separating source from derived shape). Backend-rules: "Must not introduce abstractions without clear need."

`OrgContext` (public `record : IAiContext`) exposes the eight fields generated content needs to ground itself and the renderer needs for compliance footers: `OrganisationName`, `IndustryKey`, `SubindustryKey`, `AddressLine1`, `AddressLine2`, `City`, `CountryCode`, `CountryName`. There is no renderer-only / prompt-only split — every field is plain text safe to inject into the prompt and also useful for deterministic renderers (CAN-SPAM / CASL footers need the physical address). Scope §2.14's renderer-only invariant applies to brand visual fields (colors / fonts / logos), not to org text fields. `IndustryKey` and `SubindustryKey` are passed through verbatim; the industry template lookup against `industry_template_configs` is `IndustryContext`'s job (Task 1.4.4) — `OrgContext` stays at identity level. `ToPromptString()` renders all present fields under a `## Organisation` header: address segments compose with comma joins skipping blank parts (so no trailing commas appear when `AddressLine2` or `City` is absent), and country renders as `Name (CODE)` when both are present, falls back to whichever single value is available, or is omitted entirely when neither is set.

`OrgContextBlock : IContextBlock<OrgContext>` validates `accountId != Guid.Empty` (scope §7.2) before opening the DbContext, then queries `context.Accounts.AsNoTracking().Include(a => a.Country).FirstOrDefaultAsync(a => a.Id == accountId, ct)` and disposes the context in a `finally`. `AsNoTracking()` because the block is read-only by definition; the `Country` include resolves the human-readable name in the same round trip instead of returning a code-only context. Best-effort by design — `Account` row missing → `OrgContext.Empty`; EF Core / factory throws → `Warning` log + `OrgContext.Empty` (a transient tenant-DB hiccup never crashes the AI call); `Account` present but every optional column null → populated record with `Exists = true` whose `ToPromptString()` still returns `""` until at least one field is filled (callers can distinguish "no row" from "sparse row" via `Exists` while still getting a clean prompt). `OperationCanceledException` always propagates so upstream cancellation terminates the load immediately rather than degrading silently. DI registration added to `ServiceCollectionExtensions.AddAiServices()`: `IContextBlock<OrgContext> → OrgContextBlock` as a singleton.

Verified with `dotnet build src/backend/Core/SSP.Core.sln` clean (0 errors; warnings unchanged from the pre-existing baseline) and **185/185** AI unit tests green — up from 170 with fifteen new org tests across two test classes. `OrgContextTests` (8 tests) pins the typed shape: empty stub invariants, `IAiContext` implementation, full-render path under the `## Organisation` header, partial render (only Name set, other lines omitted), address composition skipping blank segments to avoid trailing commas, country code-only / name-only / `Name (CODE)` branches, and the "exists but blank" → `""` invariant. `OrgContextBlockTests` (7 tests) exercises the loader against an in-memory `TenantDbContext`: `Guid.Empty` throws `ArgumentException` and the factory's `CreateForAccount` is **not** called (test factory throws if it is — asserts validation fires first); missing account row → empty stub with the factory call counted exactly once; populated `Account` + `Country` rows flow every field through and the rendered prompt contains `Ireland (IE)`; account row whose `CountryCode` references a missing `Country` reference row still returns the populated context with `CountryName == null` and prompt `Country: IE`; factory throws → `Warning` + empty stub; pre-cancelled token propagates as `OperationCanceledException`; `options: null` is accepted. The org portion of the composed email-generate prompt is now unblocked; the full composition still requires Task 1.4.4 (Industry).

## Task 1.4.4 — Industry Context Block

Third concrete context block built on the Task 1.4.1 base and the last block required by the PR 9 email-generate proof. ## Task 1.3.3 — Spend Cap Enforcement

Closed the cost-control gap in Story 1.3 by adding a pre-provider daily spend cap that short-circuits the AI client pipeline before any token-consuming work runs. The new `IAiSpendCapEnforcer` seam — default implementation `DailyAiSpendCapEnforcer` — sits between model resolution and content filtering in `AiClientBase.GenerateAsync`. It sums the current UTC day's `ssp.ai_usage.cost_usd` for the requesting account, compares against the effective cap, and returns a typed `AiSpendCapResult` carrying the resolved cap, current spend, and a block / allow verdict. The effective cap resolves in this order: per-account override via the new `Account.DailyAiSpendCapUsd` column (added by EF migration `20260611112835_AddAccountDailyAiSpendCap`), then `Ai:SpendCap:DefaultDailyLimitUsd`, then disabled when the section is omitted. A configurable unlimited sentinel (cap = 0 or negative) bypasses the check entirely so internal tools and load tests keep working without touching code.

Model resolution intentionally runs **before** the cap check so blocked calls still log the concrete provider and model the request would have hit — the resulting `ssp.ai_usage` row carries `success = false`, `error_message = "SpendCapExceeded: <current>/<cap> USD"`, zero tokens, and zero cost so capped attempts stay auditable and surface in the same cost rollups Story 1.3 powers. When the cap is exceeded, the pipeline throws `AiClientException` with the new `AiClientFailureKind.SpendCapExceeded` value — the typed kind lets BFF callers map the failure to a 429-equivalent without parsing the error message. The provider transport is never invoked on a blocked call (asserted by tests). A read failure on `ai_usage` (DB hiccup) fails open with a `Warning` log: a tenant-DB blip should not block AI calls for the whole org.

Verified with `dotnet build src/backend/Core/SSP.Core.sln` clean and **~233 AI unit tests** green — `DailyAiSpendCapEnforcerTests` covers default cap, per-account override, UTC current-day window boundary, disabled-section behavior, unlimited sentinel, and fail-open on DB error; `AnthropicAiClientTests` and `AzureFoundryAiClientTests` both pin the pre-provider-block invariant (no transport call when capped) and the failed-zero-cost usage row. The check is provider-agnostic — both adapters inherit the behavior through `AiClientBase`, so the Anthropic-direct → Azure AI Foundry migration does not need to re-wire spend caps.
